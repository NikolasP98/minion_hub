# Phase 7 — UI Design Contract: Validation UX

> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.

## Design System

| Property | Value |
|----------|-------|
| Tool | none — custom CSS-variable token system |
| Preset | not applicable |
| Component library | none — bespoke Svelte components with CSS custom properties |

## Spacing Scale

Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gap in validation rows, close button padding |
| sm | 8px | Gap between validation findings, panel section gaps |
| md | 16px | Panel header/footer horizontal padding (matches existing modal: 1.25rem ≈ 20px), section padding |
| lg | 24px | Panel default width rhythm reference (280px panel = 17.5 × 16px grid) |
| xl | 32px | Major layout gaps in book spread |
| 2xl | 48px | (not used in this phase) |
| 3xl | 64px | (not used in this phase) |

Source: existing page.svelte styles (modal padding: 0.875rem / 1.25rem / 0.75rem map to sm, md, sm rhythm).

Exceptions:
- Toolbar button padding: 0.375rem 0.625rem (6px × 10px) — existing non-4-grid exception inherited from `.toolbar-btn`. This phase does not introduce or extend this pattern.
- Validation panel fixed width: 280px — not a spacing token, it is a layout dimension
- Panel header height: 2.75rem (44px) — matches toolbar height for visual alignment

## Typography

All values are inherited from the existing codebase. Do not introduce new type scales.

Two weights only: 400 (regular) and 600 (emphasis). Weight 700 is not used in new components introduced by this phase.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body (validation message text) | 12px (0.75rem) | 400 | 1.5 |
| Label (chapter group header, panel title) | 11px (0.6875rem) | 600 | 1.4 |
| Micro (summary counts, passing count footer) | 11px (0.6875rem) | 400 | 1.4 |
| Section heading (panel "Validation — N errors" header) | 14px (0.875rem) | 600 | 1.2 |

Source: existing validation modal styles in +page.svelte — `.validation-title` (0.875rem), `.validation-summary` (0.6875rem), `.validation-row` (0.75rem). The existing `.validation-title` style declares weight 700; new panel header uses 600 to stay within the 2-weight contract. The executor must not add a third weight.

No new font sizes are introduced. The panel re-uses the exact scale already present in the outgoing modal.

## Color

All colors are CSS custom properties resolved at runtime from the active theme preset. No hardcoded hex values in component styles — always use the CSS variable with a fallback for the default "New York" theme.

| Role | CSS Variable | Fallback | Usage |
|------|--------------|----------|-------|
| Dominant (60%) | `--color-bg` | #09090b | Panel background surface |
| Secondary (30%) | `--color-bg2` | #18181b | Panel header/footer backgrounds, finding row backgrounds |
| Accent (10%) | `--color-accent` | #3b82f6 | Fix button foreground, focus rings on panel interactive elements |
| Error semantic | `--color-error` / `--color-destructive` | #ef4444 | Error finding rows, error icon, validation button error state, disabled-button tooltip indicator |
| Warning semantic | `--color-warning` | #f59e0b | Warning finding rows, warning icon, validation button warning state |
| Success semantic | `--color-success` | #22c55e | Publish success toast, all-clear validation button state |
| Muted | `--color-muted` | #a1a1aa | Chapter group sub-labels, passing count text, close button |
| Border | `--color-border` | #27272a | Panel border, finding row separators |

Source: `src/app.css` `@theme` block. Confirmed present: `--color-error` does not exist as a token (the codebase uses `--color-destructive` for the semantic token but also uses `--color-error` as a fallback-pattern alias in component styles). Use `var(--color-destructive, #ef4444)` consistently for errors.

Accent reserved for:
- Fix button text/icon foreground
- Focus ring on panel's interactive elements (close button, Fix buttons)
- Required field asterisk in condition editor (already established)

Accent is NOT used for: validation row backgrounds, error/warning indicators (those use semantic colors), panel border.

## Component Contract

### ValidationPanel

New component: `src/lib/components/builder/ValidationPanel.svelte`

**Layout:** Fixed-width right-side panel. The `.dag-page` section becomes `display: flex; flex-direction: row`. ChapterDAG takes `flex: 1; min-width: 0`. ValidationPanel has `width: 280px; flex-shrink: 0` when visible, `display: none` when `showValidation` is false.

**Panel open/close transition:** `transition: width 0.2s ease, opacity 0.15s ease`. When closed: `width: 0; opacity: 0; overflow: hidden`. This avoids layout jump without needing a JS animation library.

**Panel width persistence:** Write `showValidation` panel width to `localStorage` under key `validation-panel-width`. Default: 280px. Min: 220px. Max: 480px. (Width resize is at executor discretion — see RESEARCH.md Pitfall 2.)

**Panel structure:**

```
┌──────────────────────────────────────┐  ← border-right: 1px solid --color-border (panel left edge)
│ HEADER                               │  ← background: --color-bg2, height: 2.75rem, padding: 0 md
│ "Validation — N errors, N warnings"  │  ← 14px/600, color: --color-foreground
│                                  [×] │  ← close button, color: --color-muted
├──────────────────────────────────────┤
│ BODY (scrollable)                    │  ← overflow-y: auto, padding: sm md
│  [Skill group if skill-level issues] │
│    ⚠ Skill has no custom name        │  ← warning row, 12px/400
│  [Chapter 1 header]                  │  ← 11px/600 uppercase muted, with [Fix →] btn
│    ✕ Missing instructions            │  ← error row
│    ✕ No tools assigned               │  ← error row
│  [Chapter 2 header]                  │
│    ⚠ No output definition            │  ← warning row
├──────────────────────────────────────┤
│ FOOTER                               │  ← background: --color-bg2, padding: sm md
│ "N chapters passing"                 │  ← 11px/400 muted
│          [Publish Anyway] (optional) │  ← only when publishAnyway=true, accent bg
└──────────────────────────────────────┘
```

### Publish Button (modified)

**Disabled state contract:**

```css
.toolbar-btn:disabled {
  opacity: 0.5;
  cursor: not-allowed;
}
```

- `disabled` attribute set when `skillEditorDerived.validationCounts.errors > 0 || skillEditorState.publishing`
- `title` attribute: `"Fix N error(s) before publishing"` when errors > 0 (N = actual count, pluralized)
- `title` attribute when no errors: existing strings ("Publish to shared space" / "Republish with latest changes")

**onclick handler:** When errors = 0 and warnings > 0: set `skillEditorState.showValidation = true`, set `skillEditorState.publishAnyway = true`, return early. When errors = 0 and warnings = 0: call `publishSkill()` directly.

### Validation Toolbar Button (unchanged visual contract, behavior update)

Existing `.validation-btn` with `.error` / `.warning` / `.ok` modifier classes — keep identical. Toggle behavior changes from modal to panel: `onclick={() => skillEditorState.showValidation = !skillEditorState.showValidation`.

### Toasts (replacing publish-error banner)

Positions and durations from existing `toast.svelte.ts`:

| Event | Function | Title | Description | Duration |
|-------|----------|-------|-------------|----------|
| Publish success | `toastSuccess()` | "Skill published!" | (none) | 4000ms (default) |
| Server/HTTP error | `toastError()` | "Publish failed" | Error message or "Could not reach server. Please try again." | 5000ms (override: `{ duration: 5000 }`) |

Toast placement: `top` (existing `toaster` configuration — do NOT change).

### Accessibility Contract (A11Y-01)

Three modals need `role="dialog"` moved from backdrop div to inner modal div:

| Modal | Backdrop element | Modal element | Title element id |
|-------|-----------------|---------------|-----------------|
| Delete confirmation | `.confirm-overlay` | `.confirm-modal` | `delete-modal-title` |
| Condition editor | `.confirm-overlay` | `.condition-modal` | `condition-modal-title` |
| Chapter editor | (in ChapterEditor.svelte) | inner modal div | `chapter-editor-title` |

ValidationPanel uses `role="complementary"` with `aria-label="Skill validation"` — NOT role="dialog".

## Interaction Contract

### Panel Open Triggers

1. User clicks "Validation" toolbar button → toggle `showValidation` boolean
2. User clicks "Publish" with errors present → button is disabled, no action fires
3. User clicks "Publish" with warnings but no errors → `showValidation = true`, `publishAnyway = true`, publish is deferred

### Panel Close Triggers

1. User clicks "×" close button in panel header
2. User clicks "Validation" toolbar button again (toggle)
3. User opens a chapter editor (panel stays OPEN — chapter editor overlays on top, panel remains behind it)

Note: panel does NOT close on Escape key (unlike the old modal). ESC closes the chapter editor overlay only.

### Finding Reactivity

Findings update reactively via `$derived` from `skillEditorDerived.validationFindings`. No manual refresh needed. When chapter editor closes, findings re-derive automatically.

### "Publish Anyway" Visibility

`publishAnyway` flag is `true` only when: user attempted publish → warnings-only path → panel auto-opened. The flag resets to `false` when:
- Publish completes (success or error)
- User closes the panel
- User navigates away (cleanupSkillEditor)

The "Publish Anyway" button ONLY appears in the panel footer when `publishAnyway === true`.

### Disabled Button Behavior

`cursor: not-allowed` must be applied even though the button is `disabled` (pointer events are suppressed on disabled elements by default — apply cursor via CSS `:disabled` selector on the button itself, not a wrapper). The `title` tooltip renders via browser native `title` attribute — consistent with existing save indicator tooltip pattern.
