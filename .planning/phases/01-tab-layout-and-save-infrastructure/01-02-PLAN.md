    - beginRestart() sets phase to 'restarting', records startedAt, starts 30s timeout
    - If 30s pass without reconnect, phase transitions to 'failed'
    - onRestartReconnected() clears timeout, sets phase to 'reconnected', auto-transitions to 'idle' after 3s
    - If isDirty is true when onRestartReconnected is called, hadLocalChanges is set to true
    - save() calls beginRestart() when WS error includes 'closed' or 'not connected' AND a config.patch was successfully sent (i.e., the error came from the post-save loadConfig, not from the save itself)
    - discard() or loadConfig() resets restartState to idle
  </behavior>
  <action>
    **1. config.svelte.ts -- Add restart state machine:**

    Add the following exports:
    ```typescript
    export type RestartPhase = 'idle' | 'restarting' | 'reconnected' | 'failed';

    export const restartState = $state({
      phase: 'idle' as RestartPhase,
      startedAt: null as number | null,
      timeoutId: null as ReturnType<typeof setTimeout> | null,
      hadLocalChanges: false,
    });

    export function beginRestart() { ... }       // as per RESEARCH.md pattern
    export function onRestartReconnected() { ... } // checks isDirty, sets hadLocalChanges
    export function resetRestartState() { ... }    // cleanup
    ```

    Modify `save()`: Currently when WS error includes 'closed'/'not connected', it sets saveError text. Change this: if the `config.patch` sendRequest succeeded but the subsequent `loadConfig()` fails with closed/not connected, call `beginRestart()` instead of just setting saveError. The key distinction: if sendRequest itself throws 'closed', the save may not have reached the gateway (network issue). If sendRequest succeeds but the connection drops shortly after, the gateway likely restarted to apply changes. Track this with a `saveSucceeded` flag inside `save()`.

    Modify `loadConfig()`: Before replacing `configState.original` and `configState.current`, check if `restartState.phase === 'restarting'` AND `isDirty.value` is true. If so, stash the current dirty paths and values. After loading, set `restartState.hadLocalChanges = true` so the UI can show a merge warning. For the merge warning itself: use a simple confirm() dialog -- "Gateway config was reloaded after restart. You had unsaved changes. [Keep my changes] [Use gateway values]". If user keeps their changes, re-apply the stashed dirty values onto the new `configState.current`.

    **2. config.test.ts -- Unit tests:**

    Create `src/lib/state/config.test.ts` with vitest tests covering the behaviors listed above. Since `$state` runes can't run outside Svelte context in unit tests, test the state machine logic by extracting the transition functions as pure functions that take state and return new state, then test those. Alternatively, use vitest's `vi.useFakeTimers()` to test timeout behavior.

    Test cases:
    - `beginRestart` sets correct initial state
    - After 30s timeout, phase becomes 'failed'
    - `onRestartReconnected` clears timeout and transitions to 'reconnected'
    - After 3s, 'reconnected' auto-transitions to 'idle'
    - `hadLocalChanges` is set when dirty at reconnect time

    **3. ConfigSaveBar.svelte -- Add restart state UI:**

    Import `restartState` from config.svelte.ts. Add conditional rendering:

    - **restartState.phase === 'restarting':** Replace save bar content with: spinner (animated border circle) + "Restarting gateway... Reconnecting" text. Add pulsing accent left border (2px, animated opacity). Disable Discard/Save buttons.

    - **restartState.phase === 'reconnected':** Show green success state: checkmark icon + "Changes applied" text. Green left border. Auto-dismisses (the state transitions to idle after 3s, which hides the bar if not dirty).

    - **restartState.phase === 'failed':** Show error state: warning icon + "Reconnection failed" text. Red left border. Add a "Retry" button that calls `wsConnect()` (import from gateway.svelte.ts -- or just reload the page).

    - **restartState.hadLocalChanges:** If true and phase is 'reconnected', show an additional amber warning line: "You had unsaved changes that were preserved" with a small "Review" link/button.

    - **Normal (idle) state:** Keep existing save bar behavior exactly as-is.

    Style all states using CSS variables (var(--accent), var(--destructive), var(--success) or green color). Keep the slide-up animation.

    **4. gateway.svelte.ts -- Hook restart detection into onHelloOk:**

    Import `restartState, onRestartReconnected` from config.svelte.ts. In the `onHelloOk` function, add a check at the beginning:

    }

    - **beforeunload guard:** In `onMount`, add `window.addEventListener('beforeunload', handler)` where handler calls `e.preventDefault()` when `isDirty.value` is true. Clean up in the returned destructor.

    Note: Use `onMount` (not `$effect`) for the window event listeners to ensure proper cleanup.

    - Add a `data-group-id={group.id}` attribute to the root element of each ConfigSection. This is what the scrollspy's IntersectionObserver will observe.

    **2. SettingsScrollspy.svelte -- Floating right-edge indicator:**

    Create a scrollspy component that tracks which config group header is currently visible.

    Props:
    ```typescript
    let { groups, dirtyGroupIds, scrollContainer }: {
      groups: ConfigGroup[];
      dirtyGroupIds: Set<string>;
      scrollContainer: HTMLElement | null;
    } = $props();
    ```

    Behavior:
    - Uses `IntersectionObserver` to watch elements with `data-group-id` attributes inside the scroll container.
    - Observer config: `{ root: scrollContainer, rootMargin: '-10% 0px -80% 0px', threshold: 0 }` -- this means the "active" group is the one whose header is in the top 10-20% of the viewport.
    - Tracks `activeGroupId` in `$state`.
    - Renders as a fixed/absolute-positioned column on the right edge of the tab panel. Position: `right: 8px; top: 50%; transform: translateY(-50%)`.
    - For each group: render a small dot (6px circle). Active dot is accent-colored and slightly larger (8px). Inactive dots are `bg-muted-foreground/30`.
    - Dirty groups get an additional accent-colored ring or different colored dot (use var(--accent) with opacity).
    - On hover over a dot: show the group label as a tooltip (simple CSS tooltip with `::after` pseudo-element or a small absolutely-positioned span that appears on hover).
    - On click: smooth-scroll to the group element using `document.getElementById('group-' + groupId)?.scrollIntoView({ behavior: 'smooth', block: 'start' })`.
    - Semi-transparent overall: `opacity: 0.6` on container, `opacity: 1` on hover.
    - Hidden on mobile/tablet: `hidden xl:flex` (use 1024px breakpoint via Tailwind's `xl:` or `lg:` -- the CONTEXT.md says < 1024px).

    Lifecycle:
    - Set up the IntersectionObserver in `$effect` that depends on `scrollContainer` and `groups`. Disconnect old observer when deps change.
    - Use `onDestroy` or the `$effect` cleanup to disconnect the observer.

    Use CSS variables for all colors (var(--accent), var(--muted-foreground), etc.) to ensure theming consistency (PLSH-01).
